## Architecture Migration Summary

### Before (Static-First Era)

- **Hosting:** GitHub Pages (free CDN)
- **Storage:** localStorage (browser) → GitHub file (eventual)
- **Backend:** None (pure static)
- **Database:** User-configured GitHub repo
- **API:** GitHub REST v3 (`PUT /repos/.../contents/`)
- **Auth:** GitHub PAT in localStorage (single-user)

### After (Managed Backend Era)

- **Hosting:** Azure Static Web Apps (free tier, CDN + serverless)
- **Storage:** CosmosDB via DAB REST API (immediate + durable)
- **Backend:** Data API Builder (serverless, auto-generated endpoints)
- **Database:** CosmosDB container `discs` (partition key: `/manufacturer`)
- **API:** DAB REST (`POST /api/Disc`, `PUT /api/Disc/id/{id}`, `DELETE /api/Disc/id/{id}`)
- **Auth:** StaticWebApps provider (GitHub OAuth via `authenticated` role)

## Architectural Decisions Reviewed

### 1. **Primary Storage is Now CosmosDB via DAB**
- **Status:** Active (shipped)
- **Why:** Durable, multi-region capable, eliminates localStorage-clear data loss
- **Trade-off:** Requires backend deployment + CosmosDB instance (breaks GitHub Pages simplicity)
- **Fallback:** DAB downtime → localStorage fallback (offline resilience preserved)

### 2. **GitHub Sync Becomes Optional, Async Backup**
- **Status:** Active (but under review)
- **Why:** Preserve data ownership + audit trail in GitHub (shadow backup)
- **Trade-off:** Dual-source complexity (DAB + GitHub can diverge)
- **Future:** Either promote as explicit "backup" button or deprecate entirely

### 3. **Hosting: Azure SWA Replaces GitHub Pages**
- **Status:** Active
- **Why:** Single deployment target, DAB integrated, no separate Functions needed
- **Trade-off:** Vendor lock-in to Azure, more opaque infrastructure
- **Implications:** CNAME still valid, SSL auto-managed, no GitHub Actions deploy needed

### 4. **Authentication: GitHub OAuth via SWA**
- **Status:** Active (security-hardened)
- **Why:** Zero-config for SWA, single user (no multi-user complexity)
- **Trade-off:** Inaccessible to users without GitHub account
- **Future:** Path to multi-user via SWA auth rules + role-based DAB permissions

### 5. **Data Schema Unchanged**
- **Status:** Active
- **Why:** Keeps frontend stable, backward-compatible with old data
- **Known debt:** `addedAt` (ISO string from API) vs. `added` (timestamp int) in UI

## Roadmap Items (From Decision Inbox)

### 1. **Offline Resilience & Sync Strategy** (1–2 sprints)
- **Problem:** DAB down → fallback to stale localStorage, offline edits lost on reload
- **Solution:** Client-side pending queue (IndexedDB/sessionStorage), reconcile on restore
- **Risk:** Without this, offline edits silently disappear

### 2. **GitHub Sync Consistency Handling** (1 sprint)
- **Problem:** Dual-source (DAB + GitHub) can diverge silently
- **Solution:** Promote GitHub sync to explicit "Backup to GitHub" button, separate UI flow
- **Alternative:** Deprecate GitHub sync entirely, SWA export/import only

### 3. **Multi-User / Sharing Variant** (future, Phase 2)
- **Problem:** Current DAB has `anonymous` permissions (wide open)
- **Opportunity:** Add SWA auth (Azure AD, GitHub OAuth), per-user data isolation, bag list modal
- **Scope:** Major feature, schema upgrade, auth flow, UI redesign

## Known Issues / Tech Debt

1. **Timestamp mismatch:** UI `added` (ms int) vs. API `addedAt` (ISO string) — implicit conversion
2. **GitHub PAT in localStorage:** Not used in happy path, but security debt if GitHub sync continues
3. **No conflict resolution:** If DAB + GitHub both sync, GitHub falls behind silently
4. **Anonymous access:** DAB needs IP allowlist or SWA auth before production multi-tenant
