## Diagnoses

### Bug #1: Entity Source Format (GraphQL Type Mapping)
**Issue:** DAB GraphQL schema expected entity source as object with `type: "collection"`, not bare string.
**Root Cause:** staticwebapp.database.schema.gql listed entities with simple string: `Disc (source: "discs")`
**Fix:** Changed to object format: `Disc (source: { type: "collection", name: "discs" })`
**Impact:** All GraphQL queries now resolve entity to correct CosmosDB collection; eliminates 500 errors on entity source lookup

### Bug #2: GraphQL Type Name Pluralization
**Issue:** DAB GraphQL auto-generated plural types (`Discs`, `DiscsConnection`) but schema defined singular (`Disc`).
**Root Cause:** DAB pluralization rules not aligned with schema definition.
**Fix:** Added explicit type name mappings in schema to enforce singular/plural consistency.
**Impact:** GraphQL queries now correctly resolve list and connection types; no 404 on collection operations

### Bug #3: ID Field Not Non-Null
**Issue:** ID field was nullable (`id: ID`) but CosmosDB item IDs are always present.
**Root Cause:** Default GraphQL introspection didn't enforce non-null on CosmosDB `id` field.
**Fix:** Changed GraphQL field definition: `id: ID` → `id: ID!`
**Impact:** Mutations and queries now correctly validate ID presence; prevents null-reference errors downstream

### Bug #4: Partition Key Missing from Mutations
**Issue:** updateDisc and deleteDisc mutations did not include `_partitionKeyValue` parameter.
**Root Cause:** DAB requires explicit partition key in mutation payloads for CosmosDB; schema didn't expose it.
**Fix:** Added `_partitionKeyValue` as required parameter to both mutations (value matches user ID from session context).
**Impact:** Mutations now execute against correct partition; fixes CosmosDB write consistency errors
