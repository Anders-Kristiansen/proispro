## Work Summary

### Auth Lockdown — Require Authentication for All Routes and DAB Access (Completed)

**Objective:** Lock down all access to ProIsPro app and its CosmosDB backend behind GitHub OAuth authentication. Remove public access entirely per AK's security directive.

**Implementation:**

#### 1. DAB Config Changes (`staticwebapp.database.config.json`)

- **Disc entity permissions:** 
  - Changed `"role": "anonymous"` → `"role": "authenticated"`
  - Prevents unauthenticated reads or writes to Cosmos DB via `/data-api/graphql`
  
- **CORS origins:**
  - Changed `["*"]` → `["https://proispro.com"]`
  - Locks down API access to known production domain only
  - Wildcard was acceptable when data was public-read; now restricted for authenticated + write operations

#### 2. SWA Route Config (`staticwebapp.config.json`) — New File

Controls Azure Static Web Apps route-level authentication:

| Route | Allowed Roles | Behavior |
|-------|--------------|----------|
| `/.auth/*` | `anonymous` | Remains open — required for OAuth login flow |
| `/api/*` | `authenticated` | Locks down Azure Functions endpoints |
| `/*` | `authenticated` | Site-wide authentication gate |

- **401 override:** HTTP 302 redirect to `/.auth/login/github` (automatic GitHub OAuth)
  - Unauthenticated visitors never see error page; seamlessly redirected to login

#### 3. Auth Model

1. Unauthenticated user visits `proispro.com`
2. SWA checks session cookie → absent → responds with 401
3. 401 handler redirects to `/.auth/login/github`
4. User authenticates with GitHub → SWA issues encrypted session cookie
5. Subsequent requests carry session cookie → SWA validates → user has `authenticated` role
6. GraphQL/REST calls to `/data-api/*` carry SWA session → DAB validates `authenticated` role
7. Access granted to `Disc` entity operations (read/write)

## Rationale

- **Personal tool:** No business case for public access to disc inventory
- **DAB anonymous role:** Development convenience, not production design
- **CORS tightening:** Writes now in scope (DAB supports mutations via StaticWebApps auth)
- **GitHub OAuth:** Zero-config for SWA, consistent with GitHub-as-backend architecture
- **Operator burden:** Minimal — SWA authentication handled by platform

## Trade-offs

✅ **Pro:** Data is now private. No unauthorized reads or writes.  
✅ **Pro:** GitHub OAuth is zero-config on SWA. No additional identity provider needed.  
⚠️ **Con:** App inaccessible to users without GitHub account (acceptable: single-user personal tool).  
⚠️ **Con:** First-time experience shows GitHub OAuth screen instead of app (expected for this auth model).
