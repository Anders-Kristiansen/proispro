## Architecture Decision Summary

| Layer | Old (Azure) | New (Supabase + Pages) |
|-------|------------|----------------------|
| Hosting | Azure SWA + CosmosDB | GitHub Pages (free) |
| Database | CosmosDB NoSQL | PostgreSQL (standard) |
| API Layer | DAB (GraphQL) | Supabase REST client (JS SDK) |
| Auth | Azure SWA + GitHub OAuth | Supabase GitHub OAuth |
| Deploy | SWA + manual config | GitHub Actions (auto) |
| Data Portability | ❌ NoSQL lock-in | ✅ PostgreSQL standard |

## Key Lessons (Captured in docs/lessons-learned.md)

1. **Check deprecation before config debugging** — DAB retirement was silent (no helpful error messages, just 500s). Local SWA CLI worked fine; production runtime was gone.
2. **"Works locally" ≠ "works in production"** when tooling bundles its own runtime (SWA CLI vs. SWA cloud).
3. **Simpler stacks reduce failure surface** — Supabase + Pages is 2 services + 1 config file; Azure was 3+ services + multiple dashboards.
4. **Choose boring infrastructure for small apps** — PostgreSQL > NoSQL for this scale; GitHub Pages > SWA; RLS > server code.
